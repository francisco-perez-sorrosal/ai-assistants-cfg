The implementer deferred SE6 (path-scoping `eval-driven-verification.md` and `gpu-budget-conventions.md`) with rationale documented in `LEARNINGS.md ## Decisions Made — implementer (R3a-1, 2026-05-03)`.

**Verdict: defensible.** The architectural argument that both rules apply beyond ML training is sound:
- `eval-driven-verification.md` (1,507 chars) — encodes metric-threshold AC syntax with PASS/FAIL/WARN tolerance bands; this generalizes to performance optimization, capacity testing, RPS-targeting, and any threshold-based acceptance scenario.

Path-scoping these rules to ML-only files (e.g., `program.md`, `runs/**`) would prematurely narrow rules whose underlying invariants — "metric thresholds with tolerance bands" and "compute steps must declare their budget" — are not ML-specific. Routing the decision through an ADR is the right shape; treating it as a polish-pass edit could under-scope the rules for non-ML threshold work.

Token-budget cost: ~792 tokens combined (per prior report) is comfortably within the ~985-token headroom. Deferral is safe but should not be deferred indefinitely — track as a follow-up architectural decision. Filed informationally as Suggested S1 in this report (not blocking merge).
