## Context

The dashboard renders five kinds of visualization:

1. **Step DAGs** (10–30 nodes) — pipeline execution graph from IMPLEMENTATION_PLAN parallel-group annotations
2. **REQ→test bipartite** (~50 nodes) — traceability matrix from `traceability.yml`
3. **ADR lineage** (small DAG) — `supersedes` / `superseded_by` chains
4. **File-overlap heatmap** (matrix) — `git.change_coupling` from metrics report
5. **Architecture overview SVG** — already generated by LikeC4 → D2 → SVG pipeline; just needs embedding

The research finding compared `st.graphviz_chart`, `pyvis`, `streamlit-agraph`, `plotly`, `streamlit-mermaid`. Maintenance status is the most discriminating axis: `streamlit-agraph` last release 2023, `streamlit-mermaid` 2022 — stale.

## Decision

Stack:

- **`st.graphviz_chart`** (Streamlit-bundled, zero extra install) — primary for DAGs (Step DAG, ADR lineage). Generate DOT strings from frontmatter and step annotations.
- **`plotly`** (already a Streamlit transitive dep, zero extra install) — primary for heatmaps (`px.imshow`) and bipartite (sparse scatter or `go.Sankey`).
- **`pyvis`** (~2 MB, opt-in) — interactive fallback for any DAG where the user wants drag/zoom; embedded via `st.components.v1.html`. Not used by default; behind a per-page "Interactive view" toggle.
- **`st.components.v1.html`** (Streamlit-bundled) — embed LikeC4-generated SVGs directly, with `scrolling=True`.

Rejected: `streamlit-agraph` (stale), `streamlit-mermaid` (stale; see Mermaid ADR), `streamlit-elements` (heavy React bundle, unjustified), `bokeh`/`altair` for the bipartite (Plotly already in tree).

## Considered Options

### Option A — Graphviz + Plotly + pyvis-on-demand (chosen)

**Pros**: Two of the three primaries are already in tree (zero extra cost); `pyvis` is the only new dep and only ~2 MB; rendering quality covers all five viz needs; LikeC4 SVGs embed via Streamlit primitive.

**Cons**: `st.graphviz_chart` requires Graphviz binary on the host machine — usually preinstalled on macOS via Homebrew; need a graceful "Graphviz not found" message in the data layer. Lazy `pyvis` means a small UX inconsistency between the static and interactive DAG views.

### Option B — Plotly only (drop Graphviz)

**Pros**: One fewer dep; Plotly's Sankey + force-directed scatter can render DAGs. **Cons**: DAG layout via Plotly is significantly worse than DOT; Plotly does not auto-route edges; user readability suffers. Rejected.

### Option C — `streamlit-agraph` (vis.js wrapper)

**Pros**: Single library covers all DAGs interactively. **Cons**: Last release 2023 — staleness risk; new dep; covered by graphviz + pyvis combination.

### Option D — Native `st.altair_chart` for everything

**Pros**: Altair is bundled. **Cons**: Altair is for statistical charts; force-directed graph support is not built-in.

## Consequences

**Positive**: Net new dep is one library (`pyvis` ~2 MB). Heatmap and bipartite use `plotly` already in tree. DAGs render via Graphviz DOT, which is well-documented and stable. LikeC4 SVGs embed natively without any new tooling. The user gets static (default) and interactive (opt-in) views per DAG.

**Negative**: Two rendering paths for DAGs (graphviz-static + pyvis-interactive). Mitigation: `widgets/graph.py` exposes one function `render_dag(dot_string, *, interactive=False)` that picks the backend; pages do not branch.

**Risks accepted**: Host-system Graphviz is a soft dependency; if absent, DOT is rendered as text (acceptable degradation) and the install instructions point to `brew install graphviz`.
