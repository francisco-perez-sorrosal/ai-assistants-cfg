## Future Paths

Carried forward from prior ledger:

- **AGENTS.md cross-tool portability** (from ROADMAP §4.1, 2026-04-12) — Generator that produces an `AGENTS.md` file from Praxion's `CLAUDE.md` + skills metadata for cross-tool compatibility (Cursor, Copilot, etc.). AAIF (AGENTS.md standard) has emerged as a cross-vendor convention; Praxion's `CLAUDE.md` is Claude-specific. Low-effort, additive — does not change existing CLAUDE.md system. High-visibility for ecosystem adoption.

- **Agent Teams integration (experimental)** (from ROADMAP §5.1, 2026-04-12) — Claude Code's Agent Teams feature (peer-to-peer mailbox, direct teammate communication) could evolve Praxion's parallel execution model. The current fragment-file approach for implementer + test-engineer is indirect — agents write to files and hope the supervisor merges correctly. Agent Teams would enable direct communication about interface contracts. Plan: when Agent Teams stabilizes (currently experimental), build a proof-of-concept replacing the fragment-file pattern for implementer + test-engineer pairs. Keep document-mediated pipeline for sequential stages. Risk: experimental feature may change significantly or be deprecated.

- **MCP Gateway pattern** (from ROADMAP §5.3, 2026-04-12) — MCP Gateways are emerging as an enterprise pattern for centralized access control, audit, and routing. If Praxion targets broader adoption, a gateway layer over the MCP servers (memory, chronograph) would address security, observability, and governance. Action: research-only at this stage. Evaluate whether a gateway between Claude Code and Praxion's MCP servers would provide value (centralized auth, request logging, rate limiting). Risk: over-engineering if adoption stays individual-developer-scale. Prior timeline guess: Q3 2026 evaluation.

- **Pipeline shortcut paths** (from ROADMAP §5.4, 2026-04-12) — The current pipeline is sequential and heavyweight. For well-understood tasks, shortcuts would improve velocity: (a) **direct-to-planner** — user provides acceptance criteria inline, skip researcher and architect; (b) **partial pipeline** — "just plan" or "just implement step N" without full pipeline; (c) **resumable pipelines** — pick up from where a prior session left off using `WIP.md` state. Risk: must not compromise quality; shortcuts must be opt-in, not default. Dependency was previously ROADMAP §2.3 (coordinator burden reduction — shipped 2026-04-16).

- **Zero-duration span resolution** (from ROADMAP §5.5, 2026-04-12) — All chronograph OTel spans are ended immediately for Phoenix visibility, making duration metrics meaningless. Known design trade-off. Options to investigate: (a) Phoenix in-progress span rendering (does Phoenix support started-but-not-ended spans?), (b) two-span approach (start marker + end marker with duration attribute), (c) deferred end with background timer. Note: `dec-052` shipped duration-aware tool spans (PreToolUse/PostToolUse open-then-close pairs) which partially addresses this for the tool-span subset; agent and chain spans remain instant. Risk: high complexity. Current approach works; this is an improvement, not a fix.
